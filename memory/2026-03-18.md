### Root Cause: Double Padding + Container Class Issue

**Double Padding Problem:**
- AppLayout already applies `px-4` (16px each side)
- Pages were adding their own `px-4` on top
- Result: 32px padding per side = 64px total waste on mobile
- On 375px iPhone SE: 17% of screen width lost

**Container Class Problem:**
- Tailwind `container` adds responsive `max-width` breakpoints
- Relies on CSS `@layer` ordering with `max-w-full` override
- Fragile: browsers handle layer resolution differently with media queries
- Solution: Replaced `container mx-auto px-4` with `w-full px-4` in App.jsx

### Technical Patterns Learned

**CSS/Layout:**
- Tailwind `container` class can cause centering issues on mobile
- Always check for double padding (layout wrapper + page wrapper)
- Use `w-full` + responsive `px-*` instead of `container mx-auto`
- Add `box-border` to ensure padding included in width calculation

**Mobile Responsive:**
- 375px iPhone SE = smallest modern viewport to test
- 64px horizontal padding = 17% waste on small screens
- Always test on real device, not just Chrome DevTools
- Hard refresh required after deploy (Cmd+Shift+R bypasses cache)

**Code Review:**
- Adversarial audit found issues self-review missed
- Template literals in SQL = bad pattern even if sanitized
- Authorization checks often forgotten in new routes
- In-memory caches need size limits (LRU)

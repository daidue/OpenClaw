## HIGH: Manual ID Validation Violates DRY + Divergence Risk

**File:** `titlerun-api/src/utils/helpers.js`  
**Line:** 1-148  
**Framework:** TitleRun Anti-Pattern #2 + OWASP A04:2021 (Insecure Design)

### Issue

`idMatch()` implements custom ID validation logic instead of using the shared `@titlerun/validation` library. This violates DRY (Don't Repeat Yourself) and creates divergence risk — if validation rules change in the library, `idMatch()` won't receive updates.

```javascript
// helpers.js lines 17-49 (partial)
const idMatch = (a, b) => {
  // Early return for reference equality
  if (a === b) {
    // Duplicates validation logic from @titlerun/validation
    if (a == null) {
      throw new TypeError('idMatch: Cannot compare null or undefined values');
    }
    const typeA = typeof a;
    if (typeA === 'object') {
      throw new TypeError('idMatch: Cannot compare objects or arrays');
    }
    // ... 30+ more lines of validation
  }
  // ... another 60+ lines of duplicate validation logic
};
```

**Reference:** TitleRun Anti-Patterns document states:

> "Manual ID validation (Should Use Shared Library) — We have `@titlerun/validation` library that handles this (as of 2026-03-01). Use `normalizeId()` from the library."

### Impact

**At production scale:**
- Validation logic divergence: If `normalizeId()` rules change (e.g., new edge case discovered), `idMatch()` won't receive the fix
- Maintenance burden: Two codebases to maintain for ID validation (148 lines in helpers.js + library)
- Security risk: If library adds security hardening (e.g., new attack vector), `idMatch()` remains vulnerable
- Testing burden: Must test both implementations independently

**Current state:**
- 148 lines of manual validation in `idMatch()`
- Duplicates logic from `normalizeId()` with slight variations (string trimming behavior differs)

**After fix:**
- 20 lines (use library twice, compare results)
- Single source of truth for validation rules
- Automatic updates when library evolves
- Reduced attack surface

### Fix

Replace manual validation with library delegation:

```javascript
const { normalizeId } = require('../routes/tradeEngine');

/**
 * Compare two ID values for equality using shared validation library.
 * 
 * @param {string|number} a - First ID to compare
 * @param {string|number} b - Second ID to compare
 * @returns {boolean} True if IDs match after normalization
 * @throws {ValidationError} If either parameter is invalid
 */
const idMatch = (a, b) => {
  // Delegate validation to library (single source of truth)
  const normalizedA = normalizeId(a);
  const normalizedB = normalizeId(b);
  
  // Both null → cannot compare
  if (normalizedA === null && normalizedB === null) {
    throw new ValidationError('idMatch: Cannot compare null values');
  }
  
  // One null, one valid → not equal
  if (normalizedA === null || normalizedB === null) {
    return false;
  }
  
  // Both valid → compare normalized integers
  return normalizedA === normalizedB;
};
```

**What changed:**
- Removed 148 lines of manual validation
- Delegated to `normalizeId()` (single source of truth)
- Simplified to 20 lines (87% reduction)
- Inherits all library features: supply chain verification, security logging, specific error messages

## HIGH: Missing Authentication + Authorization Layer

**File:** All route files (not provided, but inferred from code structure)  
**Line:** N/A (architectural issue)  
**Framework:** OWASP A07:2021 (Identification and Authentication Failures)

### Issue

Route handlers (`tradeEngine.js`, `tradeAnalysisService.js`) accept IDs and caller context but do not enforce authentication or authorization. The `normalizeId()` function accepts optional `context` parameter with `userId`, `ip`, `endpoint`, but does not verify:

1. **Authentication:** Is the caller authenticated? (valid session/token)
2. **Authorization:** Can this user access this resource? (permission check)

```javascript
// tradeEngine.js line 91

  // FIX 2: HIGH - Input Pre-Validation
  // ... validation logic ...
  
  // FIX 3: HIGH - Security Logging (FIX #9: Added caller context)

    event: 'invalid_id_type',
    inputType,
    inputValue: safeValue,
    timestamp: new Date().toISOString(),

    callerUser: context.userId,  // ❌ Not verified! Could be forged
    callerEndpoint: context.endpoint,
    requestId: context.requestId,
  });

```

**The problem:** `context.userId` is logged but never validated. An attacker can send:

```javascript
POST /api/trades/123
{
  "userId": "admin-user-id"  // ❌ Forged! No authentication check
}
```

### Impact

**At production scale (1K users):**
- **Broken Access Control (A01:2021):** Any user can access any resource by guessing/enumerating IDs
- **Authentication Bypass (A07:2021):** No mechanism to verify user identity
- **Audit Trail Poisoning:** Forged `userId` in logs makes incident response impossible

**Attack scenarios:**
1. **ID Enumeration:** Attacker iterates `GET /api/trades/1` → `/api/trades/999999`, extracting all trades
2. **Impersonation:** Attacker sends requests with `userId: "victim-id"`, poisoning audit logs
3. **Privilege Escalation:** No authorization checks → regular user can access admin-only endpoints

**Current state:**
- Zero authentication enforcement
- Zero authorization checks
- Caller context accepted but not verified

**After fix:**
- JWT/session-based authentication middleware
- Role-based access control (RBAC) for authorization
- Verified user context propagated through request lifecycle

### Fix

Implement authentication + authorization middleware:

```javascript
// middleware/auth.js
const jwt = require('jsonwebtoken');
const { JWT_SECRET } = process.env;

/**
 * Authentication middleware - Verifies JWT token and extracts user.
 * Must be applied to all protected routes.
 */
function authenticate(req, res, next) {
  const authHeader = req.headers.authorization;
  
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({
      error: 'Authentication required',
      code: 'MISSING_TOKEN',
    });
  }
  
  const token = authHeader.substring(7);
  
  try {
    const payload = jwt.verify(token, JWT_SECRET);
    
    // Attach verified user to request (cannot be forged)
    req.user = {
      id: payload.userId,
      role: payload.role,
      email: payload.email,
    };
    
    // Attach audit context (verified, cannot be forged)
    req.auditContext = {
      userId: payload.userId,
      ip: req.ip,
      endpoint: req.path,
      requestId: req.id,
    };
    
    next();
  } catch (err) {
    return res.status(401).json({
      error: 'Invalid or expired token',
      code: 'INVALID_TOKEN',
    });
  }
}

/**
 * Authorization middleware - Checks if user can access resource.
 * Example: User can only access their own trades.
 */
function authorizeTradeAccess(req, res, next) {
  const tradeId = normalizeId(req.params.tradeId, req.auditContext);
  
  // Fetch trade from database
  const trade = await prisma.trade.findUnique({
    where: { id: tradeId },
    select: { userId: true },
  });
  
  if (!trade) {
    return res.status(404).json({ error: 'Trade not found' });
  }
  
  // Check authorization: user can only access their own trades
  if (trade.userId !== req.user.id && req.user.role !== 'admin') {
    return res.status(403).json({
      error: 'Forbidden: You do not have access to this trade',
      code: 'INSUFFICIENT_PERMISSIONS',
    });
  }
  
  // Attach trade to request for route handler
  req.trade = trade;
  next();
}

module.exports = { authenticate, authorizeTradeAccess };
```

**Apply middleware to routes:**

### Issue

**Why this matters:**
- Production may need higher limits (enterprise leagues with 2000+ teams)

- Under attack, operators cannot temporarily reduce limits without deployment

### Impact

- **Inflexibility:** Enterprise customer with 1500-team dynasty league cannot use the service (hardcoded limit blocks them)

2. Sends 1000 requests/sec with 999-team payloads (just under limit)
3. Operators want to reduce limit to 100 during attack
4. Required steps: Code change → CI/CD → deploy → restart (20+ minutes)
5. Attack continues for 20 minutes before mitigation

**Current state:**
- Hardcoded limits in source code
- Requires deployment to change

**After fix:**
- Environment-configurable via env vars
- Runtime adjustment without code changes

const slowClientTracker = new Map();

/**
 * Rate limit slow clients (per-IP throttling).
 * Tracks clients that repeatedly cause slow operations.
 */
function trackSlowClient(ip, duration) {
  if (!slowClientTracker.has(ip)) {
    slowClientTracker.set(ip, { count: 0, firstSeen: Date.now() });

  const tracker = slowClientTracker.get(ip);
  tracker.count += 1;
  tracker.lastDuration = duration;

    slowClientTracker.delete(ip);
    return false;
  }
  
  // Throttle after 10 slow operations in 1 minute
  return tracker.count >= 10;

### HIGH: 1

#### 1. Loose Equality in Null Check (Type Coercion Risk)
**File:** `workspace-titlerun/titlerun-api/src/routes/tradeEngine.js`
**Line:** 17
**Severity:** HIGH
**Category:** A03 - Injection (Type Confusion)

**Code:**
```javascript
if (result === null && id != null) {
```

**Impact:**
At 1K validation calls/day, `id != null` treats `undefined` same as `null` but accepts empty string `""`, `0`, `false`, `NaN` as valid input. This creates type confusion where non-null falsy values pass the check but still fail library validation, creating inconsistent error states.

**Security concern:** Type coercion in validation logic can mask input validation bypasses. An attacker providing `id=""` would pass this check but fail library validation, potentially exposing different error paths.

**Fix:**
```javascript
if (result === null && id !== null && id !== undefined) {
```

**Better fix (strict type guard):**
```javascript
if (result === null && typeof id !== 'undefined' && id !== null) {
  throw new TypeError(`Invalid ID: validation failed for input "${id}"`);
}
```

**Test case:**
```javascript
test('rejects empty string as invalid ID', () => {
  expect(() => normalizeId("")).toThrow(TypeError);
});

test('accepts null as valid input', () => {
  expect(normalizeId(null)).toBe(null);
});

test('accepts undefined as valid input', () => {
  expect(normalizeId(undefined)).toBe(null);
});
```

const { normalizeId: normalizeIdLib } = require('@titlerun/validation');
```

**Impact:**
Trusts external library `@titlerun/validation` without version pinning or integrity validation. If this library is compromised or has a vulnerability, all ID validation is compromised.

**Mitigation:**
1. Pin exact version in package.json (not ^1.0.0, use 1.0.0)
2. Use npm audit regularly
3. Consider subresource integrity for critical libraries
4. Document library security review process

**Fix (package.json):**

    "@titlerun/validation": "1.0.0"  // exact version, no caret
  }
}
```

**File:** `workspace-titlerun/titlerun-api/src/routes/tradeEngine.js`
**Line:** 11-15
**Severity:** LOW
**Category:** Documentation / Developer Security

**Code:**
```javascript
/**
 * @param {string|number|null|undefined} id - ID to normalize
 * @returns {number|null} - Normalized ID or null
 * @throws {TypeError} - If ID is invalid
 */
```

**Impact:**
JSDoc doesn't specify WHEN TypeError is thrown vs. when null is returned. This creates ambiguity for callers.

**Fix:**
```javascript
/**
 * Normalize and validate an ID (throws TypeError on invalid input)
 * 
 * @param {string|number|null|undefined} id - ID to normalize
 * @returns {number|null} - Normalized ID (number) or null (for null/undefined input)
 * @throws {TypeError} - If ID is invalid (non-null/undefined input that fails validation)
 * 
 * @example
 * normalizeId(123)       // => 123
 * normalizeId("456")     // => 456
 * normalizeId(null)      // => null
 * normalizeId(undefined) // => null
 * normalizeId("abc")     // => throws TypeError
 * normalizeId("")        // => throws TypeError
 */
```

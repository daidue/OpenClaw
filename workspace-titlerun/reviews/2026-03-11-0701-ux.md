## HIGH: Inconsistent Severity Signaling Confuses Triage

**File:** `HEARTBEAT.md`  
**Line:** 14, 19, 27  
**Framework:** Nielsen UX (H4: Consistency and Standards)

---

### Issue

Emoji severity levels are inconsistent with actual system criticality, making rapid triage difficult.

```bash
if [ "$API_STATUS" != "healthy" ]; then
  echo "🚨 TitleRun API unhealthy: $API_STATUS"  # CRITICAL emoji
fi

if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"  # WARNING emoji
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "Details: $SCRAPER_DETAILS"
fi

if [ "$DB_STATUS" != "connected" ]; then
  echo "🚨 TitleRun Database error: $DB_STATUS"  # CRITICAL emoji
fi
```

---

### Impact

**At operational scale (Jeff processing 3 business unit heartbeats):**
- Database down = entire app unusable (truly 🚨 CRITICAL)
- API unhealthy = might be transient restart, not always critical
- Scraper unhealthy = background job issue, rarely urgent (⚠️ is correct)

**Confusion created:**
- Jeff treats API and DB failures as equal severity
- May waste time investigating API blips while DB is actually dying
- Inconsistent mental model: "Is 🚨 always critical or sometimes just unhealthy?"

**Current state:**
- 2/3 failure types use 🚨, forcing Jeff to read details to triage
- No way to quickly identify "which failure needs immediate action?"

**After fix:**
- Severity mapping: 🚨 = service down, ⚠️ = degraded, ℹ️ = transient
- Jeff can triage by emoji in <1 second scan

---

### Fix

Map emojis to actual operational severity:

```bash
if [ "$API_STATUS" != "healthy" ]; then
  # API can be degraded but still serving traffic - use ⚠️
  echo "⚠️ TitleRun API degraded: $API_STATUS"
fi

if [ "$SCRAPER_STATUS" != "healthy" ]; then
  # Scraper is background job, low urgency - use ℹ️
  echo "ℹ️ TitleRun Scraper degraded: $SCRAPER_STATUS"
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "Details: $SCRAPER_DETAILS"
fi

if [ "$DB_STATUS" != "connected" ]; then
  # DB down = service completely unusable - keep 🚨
  echo "🚨 TitleRun Database OFFLINE: $DB_STATUS"
fi
```

**What changed:**
- 🚨 (CRITICAL) = only for service-stopping failures (DB offline)
- ⚠️ (WARNING) = degraded but operational (API unhealthy)
- ℹ️ (INFO) = background job issues (scraper)
- Added "OFFLINE" / "degraded" language to match severity

**Severity rubric:**
- 🚨 User-facing service completely down
- ⚠️ Service degraded but functional
- ℹ️ Background job or non-urgent issue

## HIGH: Missing Aggregated Health Status Reduces Scan-ability

**File:** `HEARTBEAT.md`  
**Line:** 11-28  
**Framework:** Nielsen UX (H1: Visibility of System Status, H6: Recognition over Recall)

---

### Issue

No single-line summary of overall health forces operator to mentally aggregate 3 separate checks.

```bash
# API health (now includes scraper status)
HEALTH_JSON=$(curl -s https://api.titlerun.co/health)
API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status // "unknown"')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper // "unknown"')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database // "unknown"')

if [ "$API_STATUS" != "healthy" ]; then
  echo "🚨 TitleRun API unhealthy: $API_STATUS"
fi

if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "Details: $SCRAPER_DETAILS"
fi

if [ "$DB_STATUS" != "connected" ]; then
  echo "🚨 TitleRun Database error: $DB_STATUS"
fi
```

---

### Impact

**At operational scale (Jeff processing heartbeat every 15 minutes):**
- Must read 3 separate if-block outputs to know overall health
- When all healthy: **complete silence** (no output = user guesses "probably OK?")
- When 2/3 healthy: no quick way to know "how bad is it?"

**Cognitive load:**
- Operator must remember: "Did I see 0, 1, 2, or 3 error messages?"
- No instant answer to: "Is TitleRun up or down?"
- Violates H6 (Recognition over Recall): shouldn't need to remember individual checks

**Current state:**
- 3 separate status checks, no summary
- Silent success case (violates H1: Visibility of System Status)
- Jeff's mental model: "Let me scan for error messages... I see none... I think everything is OK?"

**After fix:**
- Single-line summary: "✅ TitleRun: ALL SYSTEMS HEALTHY (3/3)"
- Degraded state: "⚠️ TitleRun: 2/3 HEALTHY (scraper degraded)"
- Critical state: "🚨 TitleRun: OFFLINE (database down)"
- 0.5 second scan time instead of 3-5 seconds

---

### Fix

Add aggregated health summary before detailed checks:

```bash
# API health (now includes scraper status)
HEALTH_JSON=$(curl -s https://api.titlerun.co/health)
API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status // "unknown"')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper // "unknown"')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database // "unknown"')

# Count healthy systems
HEALTHY_COUNT=0
[ "$API_STATUS" = "healthy" ] && ((HEALTHY_COUNT++))
[ "$SCRAPER_STATUS" = "healthy" ] && ((HEALTHY_COUNT++))
[ "$DB_STATUS" = "connected" ] && ((HEALTHY_COUNT++))

# Aggregated status (show first for scan-ability)
if [ $HEALTHY_COUNT -eq 3 ]; then
  echo "✅ TitleRun: ALL SYSTEMS HEALTHY (3/3)"
elif [ "$DB_STATUS" != "connected" ]; then
  echo "🚨 TitleRun: OFFLINE (database down)"
elif [ $HEALTHY_COUNT -eq 2 ]; then
  echo "⚠️ TitleRun: DEGRADED (1 component unhealthy)"
else
  echo "🚨 TitleRun: CRITICAL (multiple components down)"
fi

# Detailed status (only show failures)
if [ "$API_STATUS" != "healthy" ]; then
  echo "  ⚠️ API: $API_STATUS"
fi

if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "  ℹ️ Scraper: $SCRAPER_STATUS"
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "  Details: $SCRAPER_DETAILS"
fi

if [ "$DB_STATUS" != "connected" ]; then
  echo "  🚨 Database: $DB_STATUS"
fi
```

**What changed:**
- Added HEALTHY_COUNT aggregation
- Always show top-line summary (even when healthy)
- Summary uses hierarchical logic: DB failure = OFFLINE, else degraded count
- Detailed checks indented (visual hierarchy)
- Silent success case eliminated: "✅ ALL SYSTEMS HEALTHY" confirms no issues

## HIGH: Non-Actionable Scraper Details Reduce Diagnostic Value

**File:** `HEARTBEAT.md`  
**Line:** 21-23  
**Framework:** Nielsen UX (H9: Help Users Recognize, Diagnose, and Recover from Errors)

---

### Issue

Raw JSON dump of scraper details provides no actionable diagnostic information.

```bash
if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"
  # Get scraper details if available
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "Details: $SCRAPER_DETAILS"
fi
```

---

### Impact

**At operational scale (Jeff diagnosing scraper failure):**
- `.checks.scraper` likely returns: `{"lastRun":"2026-03-11T06:45:00Z","status":"stale","errorCount":3}`
- Raw JSON is not human-readable in single-line echo output
- Jeff cannot quickly answer: "When did it last run? How stale? What error?"

**Diagnostic questions unanswered:**
1. When did scraper last successfully run?
2. How long has it been stale?
3. What's the specific error causing failure?
4. Is this transient (1 failure) or persistent (100 failures)?

**Current state:**
- Output: `Details: {"lastRun":"2026-03-11T06:45:00Z","status":"stale","errorCount":3}`
- Jeff reads JSON, manually parses timestamps, calculates staleness
- Time to diagnose: 30-60 seconds

**After fix:**
- Output: `Last successful run: 18 minutes ago | Error count: 3`
- Jeff immediately knows: "Scraper hasn't run in 18 min, 3 errors = investigate"
- Time to diagnose: 3 seconds

---

### Fix

Parse and format scraper details into human-readable diagnostic info:

```bash
if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"
  
  # Parse scraper details into actionable info
  LAST_RUN=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper.lastRun // "unknown"')
  ERROR_COUNT=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper.errorCount // 0')
  ERROR_MSG=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper.lastError // "No error message"')
  
  # Calculate staleness if lastRun available
  if [ "$LAST_RUN" != "unknown" ]; then
    LAST_RUN_EPOCH=$(date -j -f "%Y-%m-%dT%H:%M:%SZ" "$LAST_RUN" "+%s" 2>/dev/null || echo 0)
    NOW_EPOCH=$(date +%s)
    MINUTES_AGO=$(( (NOW_EPOCH - LAST_RUN_EPOCH) / 60 ))
    echo "  Last run: ${MINUTES_AGO} minutes ago"
  else
    echo "  Last run: unknown"
  fi
  
  echo "  Error count: $ERROR_COUNT"
  
  if [ "$ERROR_MSG" != "No error message" ] && [ "$ERROR_MSG" != "null" ]; then
    echo "  Last error: $ERROR_MSG"
  fi
fi
```

**What changed:**
- Parse specific diagnostic fields from JSON (lastRun, errorCount, lastError)
- Calculate human-readable staleness ("18 minutes ago" vs ISO timestamp)
- Show actionable metrics: time since last run, error frequency, actual error
- Format with indentation for readability

**Diagnostic value added:**
- Staleness = "Should I restart scraper now or wait?"
- Error count = "Is this transient (1 error) or persistent (50 errors)?"
- Error message = "What's actually failing?"

### Reference

**Cognitive framework:** Nielsen UX  
**Specific principle:** Heuristic #9 - Help Users Recognize, Diagnose, and Recover from Errors

**Nielsen guidance:**

## MEDIUM: Missing "unknown" State Handling Creates Ambiguity

**File:** `HEARTBEAT.md`  
**Line:** 13-15  
**Framework:** Nielsen UX (H5: Error Prevention, H9: Error Recovery)

---

### Issue

When health endpoint returns "unknown" status, script treats it as silent failure with no diagnostic guidance.

```bash
API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status // "unknown"')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper // "unknown"')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database // "unknown"')

if [ "$API_STATUS" != "healthy" ]; then
  echo "🚨 TitleRun API unhealthy: $API_STATUS"  # Could be "unknown"
fi
```

---

### Impact

**At operational scale (health endpoint temporarily unreachable):**
- Health check timeout/error = all statuses return "unknown"
- Script outputs: "🚨 TitleRun API unhealthy: unknown"
- Jeff cannot distinguish:
  - Is API actually down? (critical)
  - Is health endpoint unreachable? (API might be fine)
  - Is response malformed? (monitoring bug)

**Ambiguity created:**
- "unknown" could mean:
  1. API is down (take action)
  2. Health endpoint is down (fix monitoring)
  3. Transient network blip (ignore)
  4. API hasn't initialized yet (wait)

**Current state:**
- "unknown" treated same as "unhealthy" = false positive alerts
- No guidance on whether to investigate API or monitoring
- Operator must manually check API to disambiguate

**After fix:**
- Explicit handling of "unknown" state
- Separate emoji (❓) signals "can't determine health"
- Diagnostic: "Health check failed - verify monitoring endpoint"

---

### Fix

Differentiate "unknown" state from "unhealthy" state:

```bash
API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status // "unknown"')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper // "unknown"')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database // "unknown"')

# Handle unknown states explicitly
if [ "$API_STATUS" = "unknown" ]; then
  echo "❓ TitleRun API status unknown - health check failed"
  echo "  Verify: https://api.titlerun.co/health endpoint is reachable"
elif [ "$API_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun API degraded: $API_STATUS"
fi

if [ "$SCRAPER_STATUS" = "unknown" ]; then
  echo "❓ TitleRun Scraper status unknown - health check failed"
elif [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "ℹ️ TitleRun Scraper degraded: $SCRAPER_STATUS"
  # ... scraper details
fi

if [ "$DB_STATUS" = "unknown" ]; then
  echo "❓ TitleRun Database status unknown - health check failed"
  echo "  Verify: Database connection from API pod"
elif [ "$DB_STATUS" != "connected" ]; then
  echo "🚨 TitleRun Database OFFLINE: $DB_STATUS"
fi
```

**What changed:**
- Explicit `if [ "$X" = "unknown" ]` check before unhealthy check
- Unique emoji (❓) for monitoring failure vs component failure
- Actionable diagnostic: "Verify health endpoint reachable"
- Distinguishes monitoring bug from actual service failure

**Decision tree:**
- unknown = ❓ (monitoring issue - check health endpoint)
- unhealthy/error = 🚨/⚠️ (component issue - investigate service)

## MEDIUM: Silent Success Case Violates Visibility Principle

**File:** `HEARTBEAT.md`

**Framework:** Nielsen UX (H1: Visibility of System Status)

---

### Issue

When all systems healthy, script produces no output, forcing operator to infer "no news = good news."

```bash
if [ "$API_STATUS" != "healthy" ]; then
  echo "🚨 TitleRun API unhealthy: $API_STATUS"
fi

if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')

  echo "🚨 TitleRun Database error: $DB_STATUS"

# When all healthy: NO OUTPUT (silent success)
```

---

### Impact

**At operational scale (Jeff processing heartbeat log):**
- All healthy = 0 lines of output
- Jeff must infer: "I see no TitleRun output... did check run? Or skip? Or timeout?"
- Cannot distinguish:
  1. All systems healthy (good)
  2. Health check didn't run (bad)
  3. Script crashed before checks (bad)
  4. Network timeout prevented checks (bad)

**Ambiguity:**
- Silence could mean success OR failure
- Operator cannot confirm: "TitleRun check actually executed"
- Violates Nielsen H1: system should actively communicate status

**Current state:**
- Healthy heartbeat: 0 lines
- Jeff mental model: "I think TitleRun is OK because I don't see errors?"
- Confirmation anxiety: "But did the check actually run?"

**After fix:**
- Healthy heartbeat: "✅ TitleRun: ALL SYSTEMS HEALTHY (3/3)"
- Jeff mental model: "TitleRun is confirmed healthy"
- Zero ambiguity

**Note:** This finding overlaps with Finding #2 (Aggregated Health Status), which already provides the fix. Including as separate finding to document the specific anti-pattern (silent success).

---

### Fix

Always output health status, even when healthy:

```bash
# Always provide positive confirmation when healthy
if [ "$API_STATUS" = "healthy" ] && [ "$SCRAPER_STATUS" = "healthy" ] && [ "$DB_STATUS" = "connected" ]; then
  echo "✅ TitleRun: ALL SYSTEMS HEALTHY (3/3)"
else
  # Failure cases with detailed output
  if [ "$API_STATUS" != "healthy" ]; then
    echo "⚠️ TitleRun API degraded: $API_STATUS"
  fi
  # ... other failure checks
fi
```

**What changed:**
- Added explicit success case output
- Positive emoji (✅) signals "everything OK"
- Confirms check executed successfully
- Operator no longer needs to infer health from silence

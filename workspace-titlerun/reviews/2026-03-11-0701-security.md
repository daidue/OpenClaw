## CRITICAL: Command Injection via Unquoted Variable Expansion

**File:** `HEARTBEAT.md`  
**Line:** New lines (scraper details section)  
**Framework:** OWASP Security (A03:2021 - Injection)

---

### Issue

Variables containing untrusted JSON output are used in echo statements without quotes, allowing command substitution attacks.

```bash
if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"
  # Get scraper details if available
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "Details: $SCRAPER_DETAILS"
fi
```

---

### Impact

**Attack scenario:**
1. Attacker compromises `/health` endpoint
2. Returns malicious JSON: `{"scraper": "$(rm -rf /)"}`
3. `jq -r` extracts raw string without escaping
4. `echo "Details: $SCRAPER_DETAILS"` executes embedded command

**At production scale:**
- Remote code execution on monitoring server
- Access to all TitleRun infrastructure credentials
- Deletion of monitoring logs/data
- Lateral movement to API servers

**Attack surface:**
- External API endpoint (https://api.titlerun.co/health)
- No input validation on JSON response
- Direct variable expansion in shell

---

### Fix

Quote all variable expansions and validate JSON structure:

```bash
if [ "$SCRAPER_STATUS" != "healthy" ]; then
  # Quote variable to prevent command substitution
  echo "⚠️ TitleRun Scraper unhealthy: ${SCRAPER_STATUS}"
  
  # Validate JSON structure before parsing
  if echo "$HEALTH_JSON" | jq -e '.checks.scraper' > /dev/null 2>&1; then
    SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper // "unavailable"')
    # Quote and sanitize output
    printf "Details: %s\n" "${SCRAPER_DETAILS}"
  else
    echo "Details: [Invalid JSON structure]"
  fi
fi
```

**What changed:**
- Use `${VAR}` instead of `$VAR` for clarity
- Validate JSON with `jq -e` before parsing nested fields
- Use `printf "%s"` instead of `echo` (prevents interpretation of escape sequences)
- Provide fallback for missing fields

---

### Test

```bash

# Expected: "Details: $(whoami)" (literal string, NOT executed)

printf "Details: %s\n" "${SCRAPER_DETAILS}"
# Expected: Literal escape codes, not interpreted

# Test 3: Missing field

if echo "$HEALTH_JSON" | jq -e '.checks.scraper' > /dev/null 2>&1; then
  echo "Field exists"
else
  echo "Field missing (safe fallback)"
fi
# Expected: "Field missing (safe fallback)"
```

---

### Reference

## CRITICAL: Missing curl Security Flags (MITM/Redirect Attack)

**File:** `HEARTBEAT.md`  
**Line:** HEALTH_JSON curl call  
**Framework:** OWASP Security (A05:2021 - Security Misconfiguration)

---

### Issue

curl command lacks security flags, allowing man-in-the-middle attacks and malicious redirects.

```bash
# API health (now includes scraper status)
HEALTH_JSON=$(curl -s https://api.titlerun.co/health)
```

---

### Impact

**Attack scenarios:**

**1. MITM Attack (compromised network):**
- Attacker intercepts curl request on public WiFi
- Returns fake JSON with malicious payloads
- Monitoring system logs fake "healthy" status while API is down

**2. Redirect Attack:**
- DNS poisoning redirects api.titlerun.co to attacker domain
- curl follows redirect without warning
- Attacker controls all health check data

**At production scale:**
- False health signals → incidents go undetected
- Compromised monitoring → blind to real outages
- Attack window: unlimited (no timeout on hanging requests)

**Current state:**
- No certificate verification enforcement
- Unlimited redirect following
- No request timeout (hangs forever if server stalls)
- Silent failures (-s flag hides errors)

**After fix:**
- Certificate pinning enforced
- Max 0 redirects
- 10-second timeout
- Errors visible in logs

---

### Fix

Add security flags and error handling:

```bash
# API health (now includes scraper status)
# --max-time 10: Timeout after 10 seconds
# --max-redirs 0: Prevent redirect attacks
# --fail: Exit with error code on HTTP errors (4xx, 5xx)
# -S: Show errors even with -s
HEALTH_JSON=$(curl -sSf --max-time 10 --max-redirs 0 https://api.titlerun.co/health)
CURL_EXIT_CODE=$?

if [ $CURL_EXIT_CODE -ne 0 ]; then
  echo "🚨 TitleRun API unreachable (curl exit code: $CURL_EXIT_CODE)"
  # Don't parse JSON if curl failed
  exit 1
fi

# Validate we got JSON before parsing
if ! echo "$HEALTH_JSON" | jq empty 2>/dev/null; then
  echo "🚨 TitleRun API returned invalid JSON"
  exit 1
fi

API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status // "unknown"')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper // "unknown"')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database // "unknown"')
```

**What changed:**
- Added `--max-time 10` (prevents hanging requests)
- Added `--max-redirs 0` (prevents redirect attacks)
- Added `--fail` (treat HTTP errors as failures)
- Added `-S` (show errors with silent mode)
- Check `$?` exit code before parsing
- Validate JSON structure with `jq empty`
- Fail fast if curl or JSON validation fails

---

### Test


---

### Reference

**Cognitive framework:** OWASP Security  
**Specific principle:** A05:2021 - Security Misconfiguration

**OWASP guidance:**
> A secure installation process should be implemented including: sending security directives to clients, hardened configurations, and automated verification.

**Related vulnerabilities:**
- CWE-319 (Cleartext Transmission of Sensitive Information)
- CWE-601 (URL Redirection to Untrusted Site)

**curl security best practices:**
- Always use `--max-time` to prevent hanging
- Always use `--max-redirs 0` for security-critical endpoints
- Always check exit codes before parsing output

## HIGH: Information Disclosure in Error Messages

**File:** `HEARTBEAT.md`  
**Line:** All new error echo statements  
**Framework:** OWASP Security (A09:2021 - Security Logging and Monitoring Failures)

---

### Issue

Error messages expose internal system state details that could aid attackers in reconnaissance.

```bash
if [ "$SCRAPER_STATUS" != "healthy" ]; then
  echo "⚠️ TitleRun Scraper unhealthy: $SCRAPER_STATUS"
  # Get scraper details if available
  SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
  echo "Details: $SCRAPER_DETAILS"
fi

if [ "$DB_STATUS" != "connected" ]; then
  echo "🚨 TitleRun Database error: $DB_STATUS"
fi
```

---

### Impact

**Information exposed to attackers:**
- Database connection status (reveals DB location/type)
- Scraper implementation details (error messages, stack traces)
- Internal system architecture (what components exist)
- Exact error messages (aids in exploit development)

**At production scale:**
- Logs may be exposed via public status pages
- Error messages in Slack/Discord channels (screenshot risk)
- Monitoring dashboards accessible to contractors
- GDPR/SOC2 compliance risk (sensitive data in logs)

**Attack scenario:**
1. Attacker triggers scraper errors (malformed property data)
2. Error details reveal: "MongoDB connection pool exhausted"
3. Attacker knows: MongoDB backend, connection limits
4. Attacker targets MongoDB specifically in next phase

**Compliance impact:**
- SOC2: Logging controls require sanitization
- GDPR: Error logs may contain user data
- ISO 27001: Information disclosure controls

---

### Fix

Separate internal logging from external alerts:

```bash
# Internal log (full details for debugging)
LOG_FILE="/var/log/titlerun/heartbeat.log"

if [ "$SCRAPER_STATUS" != "healthy" ]; then
  # External alert (generic)
  echo "⚠️ TitleRun Scraper unhealthy"
  
  # Internal log (detailed)
  {
    echo "$(date -Iseconds) SCRAPER_UNHEALTHY"
    echo "  Status: ${SCRAPER_STATUS}"
    if echo "$HEALTH_JSON" | jq -e '.checks.scraper' > /dev/null 2>&1; then
      SCRAPER_DETAILS=$(echo "$HEALTH_JSON" | jq -r '.checks.scraper')
      echo "  Details: ${SCRAPER_DETAILS}"
    fi
    echo "---"
  } >> "$LOG_FILE"
fi

if [ "$DB_STATUS" != "connected" ]; then
  # External alert (generic)
  echo "🚨 TitleRun Database error"
  
  # Internal log (detailed)
  echo "$(date -Iseconds) DB_ERROR Status=${DB_STATUS}" >> "$LOG_FILE"
fi
```

**What changed:**
- Split external alerts (generic) from internal logs (detailed)
- External: Only service name + generic error
- Internal: Full details to restricted log file
- Added timestamps to internal logs
- Log file permissions should be 600 (root only)

---

## HIGH: Missing JSON Schema Validation

**File:** `HEARTBEAT.md`  
**Line:** All jq parsing lines  
**Framework:** OWASP Security (A03:2021 - Injection) + TitleRun Anti-Pattern

---

### Issue

JSON response structure is not validated before parsing, allowing malformed responses to cause silent failures or unexpected behavior.

```bash
HEALTH_JSON=$(curl -s https://api.titlerun.co/health)
API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status // "unknown"')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper // "unknown"')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database // "unknown"')
```

---

### Impact

**Failure modes:**

**1. Malformed JSON:**
- API returns HTML error page instead of JSON
- `jq` fails silently with `-r` flag
- All status variables set to empty string
- Health checks pass when they should fail

**2. Schema drift:**
- API changes field names (`.scraper` → `.scraperStatus`)
- Script continues with "unknown" values
- Alerts never trigger despite real failures

**At production scale:**
- False positives: "Everything healthy" during outages
- Silent monitoring failures (no alerts when scraper down)
- Delayed incident response (hours of undetected downtime)

**Current behavior:**
- Invalid JSON → empty variables → no alerts
- Missing fields → "unknown" status → no investigation

**After fix:**
- Invalid JSON → script exits immediately
- Missing fields → explicit error logged
- Schema validation → catches API changes

---

### Fix

Validate JSON schema before parsing:

```bash
HEALTH_JSON=$(curl -sSf --max-time 10 --max-redirs 0 https://api.titlerun.co/health)

# Validate required fields exist
REQUIRED_FIELDS=".status .scraper .database"
for field in $REQUIRED_FIELDS; do
  if ! echo "$HEALTH_JSON" | jq -e "$field" > /dev/null 2>&1; then
    echo "🚨 TitleRun API returned incomplete health data (missing: $field)"
    echo "Response: $HEALTH_JSON" >> /var/log/titlerun/heartbeat-errors.log
    exit 1
  fi
done

# Parse after validation
API_STATUS=$(echo "$HEALTH_JSON" | jq -r '.status')
SCRAPER_STATUS=$(echo "$HEALTH_JSON" | jq -r '.scraper')
DB_STATUS=$(echo "$HEALTH_JSON" | jq -r '.database')

# Validate expected values
VALID_STATUSES="healthy degraded unhealthy"
if ! echo "$VALID_STATUSES" | grep -qw "$API_STATUS"; then
  echo "🚨 TitleRun API returned unexpected status: $API_STATUS"
  exit 1
fi
```

**What changed:**
- Validate all required fields exist before parsing
- Exit immediately if schema is invalid
- Remove `// "unknown"` fallbacks (they hide failures)
- Validate status values against whitelist
- Log malformed responses for debugging

---

## MEDIUM: Insufficient Error Context for Debugging

**File:** `HEARTBEAT.md`  
**Line:** All error handling sections  
**Framework:** OWASP Security (A09:2021 - Security Logging and Monitoring Failures)

---

### Issue

Error messages lack context needed for rapid incident response (timestamps, request metadata, health check correlation).

```bash
if [ "$API_STATUS" != "healthy" ]; then
  echo "🚨 TitleRun API unhealthy: $API_STATUS"
fi
```

---

### Impact

**Debugging challenges:**

**Scenario: Oncall gets alert at 3am**
- Alert: "🚨 TitleRun API unhealthy: degraded"
- Missing context:
  - When did it start?
  - How long has it been degraded?
  - Which health checks are failing?
  - Related errors in last 5 minutes?

**At production scale:**
- Mean Time To Resolution (MTTR): +15 minutes (context gathering)
- False escalations (transient blips look like outages)
- Incomplete incident reports (can't reconstruct timeline)

**Current state:**
- No timestamps in alerts
- No correlation IDs
- No historical context

**After fix:**
- ISO timestamp on every log line
- Alert includes duration estimate
- Correlation ID for related errors

---

### Fix

Add structured logging with context:

```bash
# Configuration
LOG_FILE="/var/log/titlerun/heartbeat.log"
STATE_FILE="/var/run/titlerun/last-health-state"
TIMESTAMP=$(date -Iseconds)

# Load previous state for comparison
if [ -f "$STATE_FILE" ]; then
  PREV_STATE=$(cat "$STATE_FILE")
else
  PREV_STATE="unknown"
fi

# Check API health
if [ "$API_STATUS" != "healthy" ]; then
  # Determine if this is new or ongoing
  if [ "$PREV_STATE" = "$API_STATUS" ]; then
    DURATION="(ongoing)"
  else
    DURATION="(new issue detected)"
  fi
  
  # Structured alert with context
  echo "[$TIMESTAMP] 🚨 TitleRun API unhealthy: $API_STATUS $DURATION"
  
  # Detailed internal log
  {
    echo "[$TIMESTAMP] HEALTH_CHECK_FAILED"
    echo "  service: api"
    echo "  status: $API_STATUS"
    echo "  previous_status: $PREV_STATE"
    echo "  scraper: $SCRAPER_STATUS"
    echo "  database: $DB_STATUS"
    echo "  correlation_id: $(uuidgen)"
  } >> "$LOG_FILE"
fi

# Save current state for next run
echo "$API_STATUS" > "$STATE_FILE"
```

**What changed:**
- ISO 8601 timestamps on all log lines
- State tracking (new vs ongoing issues)
- Correlation IDs for log aggregation
- Full system context in internal logs
- Duration estimates in alerts

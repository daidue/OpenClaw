### HIGH Issues: 2

#### 1. Generic Error Message Hinders Debugging

**File:** `workspace-titlerun/titlerun-api/src/routes/tradeEngine.js:15`

**Code:**
```javascript
throw new TypeError('Invalid ID: validation failed');
```

**Impact:**  
When a developer passes an invalid ID, the error message provides NO actionable information:
- What was the invalid input? (Not shown)
- Why is it invalid? (Not explained)
- What format is expected? (Not documented in error)
- How to fix it? (No guidance)

**User frustration:**  
Developer has to:
1. Add debug logging to see what input was passed
2. Read validation library source code to understand rules
3. Trial-and-error different inputs to figure out valid format

This increases debugging time from **~1 minute** (with good error) to **~10-30 minutes** (with generic error).

**Fix:**
```javascript
if (result === null && id !== null && id !== undefined) {
  throw new TypeError(
    `Invalid ID: expected positive integer, got ${JSON.stringify(id)} (${typeof id}). ` +
    `Valid examples: 123, "456". See @titlerun/validation for rules.`
  );
}
```

**Trade-offs:**
- ⚠️ Security: Detailed errors could leak info to attackers (mitigate: only show in development)
- ✅ Debugging: Developers fix issues 10x faster

**Recommendation:**
```javascript
// Development-only detailed errors
if (result === null && id !== null && id !== undefined) {
  const message = process.env.NODE_ENV === 'production'
    ? 'Invalid ID: validation failed'
    : `Invalid ID: validation failed for input ${JSON.stringify(id)} (${typeof id})`;
  throw new TypeError(message);
}
```

**Severity:** HIGH  
**Effort:** 5-10 minutes  
**Priority:** Fix before deploy (major DX improvement)

#### 2. API Ambiguity: Null Return vs Exception

**File:** `workspace-titlerun/titlerun-api/src/routes/tradeEngine.js:11-19`

**Code:**
```javascript
function normalizeId(id) {
  const result = normalizeIdLib(id);
  
  if (result === null && id != null) {
    throw new TypeError('Invalid ID: validation failed');
  }
  
  return result;
}
```

**Impact:**  
The function has TWO different behaviors for null:
1. Input `null` → returns `null` (valid case)
2. Input `"abc"` (invalid) → library returns `null` → function throws TypeError

**Developer confusion:**
```javascript
// When does this return null vs throw?
const result = normalizeId(userInput);

// Unclear from API:
// - Is null a valid return value? (Yes)
// - Does it ever throw? (Yes, sometimes)
// - When does it throw vs return null? (Have to read docs carefully)
```

**Ambiguous contract:**
- JSDoc says: "returns `number|null`"
- JSDoc says: "throws TypeError"
- Developer thinks: "Why would it return null AND throw? Confusing!"

**Clearer API alternatives:**

**Option 1: Always throw, never return null**
```javascript
function normalizeId(id) {
  if (id === null || id === undefined) {
    throw new TypeError('ID is required');
  }
  
  const result = normalizeIdLib(id);
  if (result === null) {
    throw new TypeError('Invalid ID');
  }
  
  return result; // Always number, never null
}
```

**Option 2: Always return, never throw (Go-style)**
```javascript
function normalizeId(id) {
  const result = normalizeIdLib(id);
  return {
    value: result,
    error: (result === null && id != null) ? new Error('Invalid ID') : null
  };
}

// Usage:
const { value, error } = normalizeId(id);
if (error) { /* handle */ }
```

**Option 3: Document the contract more clearly**
```javascript
/**
 * @param {string|number|null|undefined} id
 * @returns {number|null} Returns null ONLY if input is null/undefined (passthrough).
 *                        Returns number if input is valid.
 * @throws {TypeError} Throws if input is non-null/undefined but invalid.
 */
```

**Recommendation:**  
Keep current API (for backward compatibility) but improve JSDoc:

```javascript
/**
 * Normalize and validate an ID (throws TypeError on invalid input)
 * 
 * @param {string|number|null|undefined} id - ID to normalize
 * @returns {number|null} - Normalized number, or null if input was null/undefined
 * @throws {TypeError} - If ID is invalid (e.g., "abc", NaN, negative)
 * 
 * @description
 * This function has two modes:
 * 1. Passthrough: null/undefined input → null output (no validation)
 * 2. Validation: any other input → validated number OR TypeError
 * 
 * @example
 * // Passthrough mode
 * normalizeId(null)       // → null (no error)
 * normalizeId(undefined)  // → null (no error)
 * 
 * @example
 * // Validation mode
 * normalizeId(123)        // → 123 (valid)
 * normalizeId("456")      // → 456 (valid)
 * normalizeId("abc")      // throws TypeError (invalid)
 */
```

**Severity:** HIGH  
**Effort:** 2 minutes (documentation)  
**Priority:** Update docs now, consider API redesign in next major version
